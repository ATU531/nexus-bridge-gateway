# NexusBridge 业务需求文档

## 一、业务需求列表 (Business Requirements)

### 1. 渐进式 Web3 接入 (Progressive Onboarding)

#### 1.1 统一身份转换
- 用户无需一上来就创建钱包，支持通过手机号/邮箱登录（Web2），后台静默关联或生成托管钱包（Web3）。

#### 1.2 无感签名
- 对于高频、低价值的链上交互（如领取积分），网关代为签名，提升用户体验。

### 2. 企业级资产管理与安全 (Enterprise Security)

#### 2.1 多维度风控
- 在请求发出前，自动校验目标合约地址是否为黑名单，识别恶意转账。

#### 2.2 权限细分
- 支持 Web2 的 RBAC 权限体系，例如：普通员工只能查询余额，财务主管才能发起转账。

### 3. 数据透明与对账 (Transparency & Accounting)

#### 3.1 链上事件索引
- 将复杂的异步链上事件（如充值成功）转换为标准的 Webhook 或消息队列信号，方便 Web2 业务系统对账。

#### 3.2 统一流水账单
- 将 Web2 的 API 调用记录与 Web3 的交易哈希（TxHash）关联，提供完整的全链路审计。

## 二、使用场景需求 (User Stories / Functional Requirements)

### 1. 对于前端开发者 (Frontend Dev)

#### 1.1 标准 REST/GraphQL 接口
- 开发者不需要学习 web3.js 或 ethers.js，直接通过网关提供的 HTTP 接口查询余额、调用合约。

#### 1.2 实时状态推送
- 通过 WebSocket 或长连接，网关主动推送交易状态变更（Pending -> Success），前端无需轮询。

### 2. 对于后端业务系统 (Backend Service)

#### 2.1 高并发 Nonce 管理
- 业务系统批量发奖励时，网关负责处理以太坊的 Nonce 顺序问题，防止交易冲突或覆盖。

#### 2.2 动态 Gas 策略
- 网关自动根据链上拥堵情况调整 Gas Price，确保交易在规定时间内打包，同时成本最低。

### 3. 对于运维与安全团队 (Ops & Security)

#### 3.1 监控大屏
- 直观展示网关的请求量、Gas 消耗统计、各链节点延迟状态。
- **链上 RPC 节点健康度 (Node Health)**：实时监控公共节点（如 Cloudflare）的响应延迟、错误率和限流状态，帮助运维及时发现节点异常并切换备用节点。

#### 3.2 熔断机制
- 当链上 Gas 价格异常飙升（如遇到热门项目发售）时，自动触发熔断，保护账户余额不被耗尽。

## 三、典型使用场景 (Use Cases)

### 场景 A：传统电商/平台集成 Web3 奖励

**业务逻辑**：一个普通的 App（如外卖、运动 App）想给用户发链上积分（Token）或勋章（NFT）。

**痛点**：用户没有钱包，也不懂什么是 Gas 费。

**NexusBridge 的作用**：

- **统一认证**：用户用手机号登录，网关自动在后台映射一个托管地址。
- **代付 Gas**：用户点击"领取奖励"，网关调用交易编排模块，用企业账户代付手续费。
- **异步通知**：交易成功后，网关通过 Webhook 通知 App 更新 UI。

### 场景 B：企业级 Web3 资产看板 (Dashboard)

**业务逻辑**：公司财务部门需要实时监控多个链上账户的余额和资金变动。

**痛点**：直接调公链 RPC 节点不稳定，且不同链的数据格式不统一。

**NexusBridge 的作用**：

- **核心查询引擎**：提供标准化的 REST 接口，财务系统只需输入地址，即可获得多链聚合后的资产数据。
- **大额预警**：网关监听链上事件，一旦发现异常大额转出，立即通过企业微信/钉钉报警。

### 场景 C：游戏资产的"渐进式"上链

**业务逻辑**：一款 Web2 游戏，玩家获得的装备（NFT）可以选择保存在服务器（Web2），也可以选择提取到自己的钱包（Web3）。

**痛点**：实时同步游戏数据库与链上状态非常困难。

**NexusBridge 的作用**：

- **中继器**：当玩家发起"提取"操作，网关负责锁定 Web2 数据库，同时发起链上铸造（Mint），并确保两者状态最终一致。

## 四、角色视角：谁是你的用户？

| 角色 | 他如何使用 NexusBridge？ | 获得的价值 |
|------|-------------------------|-----------|
| **前端开发者** | 通过普通的 Axios/Fetch 调用网关的 REST API。 | 无需学习复杂的 `ethers.js` 或处理私钥加密。 |
| **后端/架构师** | 将网关视为一个微服务，处理 Nonce 冲突和交易重试。 | 无需担心链上拥堵导致的交易卡死。 |
| **运维人员** | 通过网关提供的 Actuator 或监控接口查看节点延迟。 | 提高了 Web3 服务在企业内部的可观测性。 |
| **普通终端用户** | 感受不到网关的存在，只觉得自己能顺畅使用 Web3 功能。 | 极大降低了使用门槛，不再需要管理助记词。 |
